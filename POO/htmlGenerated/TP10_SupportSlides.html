3. Principe des tests

Dans ce module (et aussi en S1), de nombreux programmes de tests automatiques JUnit vous ont t proposs pour dvelopper vos classes.

Les tests automatiques permettent : de tester une unit "seule" : une classe en Java. On cherche valider le code crit pour la classe.

Les tests unitaires :

  • font une valuation partielle : une classe,
  • ne garantissent pas le fonctionnement de lintgralit du programme mais d’une classe seulement,
  • ralisent des tests d’assertions vrifier sur diffrentes situations critiques couvrant l’ensemble des situations possibles,
    • assertion : vrifier que, compte tenu de ce qu’on attend, la classe teste produit des objets dans un tat attendu.
    • assertion :
      • vrifier qu’une condition soit vraie ou fausse,
      • vrifier la valeur retourne par un observateur,
      • vrifier qu’une exception attendue se produise bien,
  • tests unitaires == suite de tests indpendants,
  • attention : possibilit de bug dans les tests !!

La mise en place de programmes de test peut tre ralise pour atteindre plusieurs objectifs, par exemple :

  1. Maintenance logicielle :
    • volution des programmes au cours du temps ⇒ empcher l’apparition de nouveaux bugs,
    • ncessit de tester aprs chaque modification ⇒ tests unitaires rpts chaque modification de classe.
  2. Concevoir/Dvelopper par les tests :
    • spcifier une classe,
    • puis commencer par crire les tests,
    • puis criture de la classe,
    • puis criture de mthodes de la classe errones qui font chouer le test : test du test.

Le couple programmeATester (classe tester) / programmeDeTest (classe de test) que l’on maintient dans le temps en quilibre peut sintgrer dans les processus industriels dintgration continue, de Tiers Maintenance Applicative (TMA) ou de contribution logicielle (par exemple libre).

4.2. Tests unitaires simples de classes

La mthodologie de test applique ici reste partielle.

L’objectif est de comprendre le principe.

Le test unitaire d’une classe consiste a minima tester l'tat attendu des objets aprs application :

  1. de chaque constructeur dont les cas de prconditions attendues,
  2. de chaque transformateur dont les cas de prconditions attendues,

Le test unitaire d’une classe consistera donc a minima crire une mthode de test (ou plusieurs) par constructeur/transformateur dans laquelle :

  1. on met un objet tester dans une situation/tat particulier,
    • des lignes de codes pour crer un objet et y appliquer si besoin des mthodes transformateur pour se mettre dans la situation dsire,
  2. vrifier ensuite que l’objet est bien dans l'tat attendu,
    • appliquer chaque accesseur pour vrifier que la valeur retourne est bien celle attendue. On teste la fois un constructeur/transformateur et les accesseurs.
