Le but est de comprendre comment mettre en place des associations navigables dans les deux sens, les traitements impliqus et les limites existantes.

En TP, nous avons dj travaill le cas de l’association oriente (navigable dans un seul sens) ou agrgation dans le cas de la classe AgenceBancaire :

  • la classe AgenceBancaire contient une ArrayList de Compte,

  • les objets Compte ne sont pas relis une AgenceBancaire.

Pour mettre en oeuvre une association, le principe sera toujours de :

  • stocker les "liens" d’une classe vers une autre par un attribut, et ce dans les deux classes,

  • selon la multiplicit : l’attribut se voit changer de type (objet, tableau/ArrayList, …),

  • il faudra crer des mthodes permettant de maintenir les relations en "tat cohrent" par rapport aux multiplicits.

2.2. Une version cohrente : maintient des liens

Pour rsoudre le problme et maintenir les liens cohrents : il faut que lors de l’association d’une Commande un Client :

  • si le Client est dj associ une Commande, l’association de cette Commande vers ce Client soit rompue,

  • si la Commande affecte est dj associe un Client, cette association soit rompue,

  • associer la Commande affecte au Client,

  • associer le Client la Commande

il en est de mme lorsqu’un Client est associ une Commande.

Question : Ecrire :

  • dans la classe Commande une nouvelle mthode void definirClient (Client cli),

  • dans la classe Client, une nouvelle mthode duale de fonctionnement similaire : public void definirCommande (Commande comm).

La mthode void definirClient (Client cli) dans la classe Commande :

  • Vrifie si le paramtre cli est null : auquel cas ne rien faire.

  • Si le Client cli est associ une Commande : dfaire cette association (associer cette Commande au Client null (setClient(null))).

  • Si la Commande est dj associe un Client : dfaire cette association (associer ce Client la Commande null (setCommande(null))).

  • Associer la Commande et le Client cli.
