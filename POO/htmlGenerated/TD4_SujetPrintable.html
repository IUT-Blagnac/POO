1. Principes de stockage de donnes

1.1. Stockage de donnes en mmoire / Possibilits du langage

Pour stocker des donnes, ce qui est donn comme disponible au programmeur en Java est :

  1. les types de base,

  2. le type tableau d'lments d’un type donn,

  3. le type "classe(/structure)" : des objets en mmoire contenant des attributs.

Pour stocker en mmoire "plusieurs" donnes, disons une "suite"/"squence"/"liste"/… de donnes, il existe deux structures de programmation de base :

  1. le tableau,

  2. la structure chane (sous forme linaire ou arborescente).

1.2. Structures de donnes

En dehors de tout contexte particulier (langage et application), une structure de donne dfinit :

  1. un ensemble d’informations organises,

  2. l’organisation de ces informations,

  3. les oprations applicables sur ces informations … et leur rsultat respectif.

Exemples: pile, file, liste, ensemble, map (clef/valeur), arbre, graphe, …

Pour les utiliser, il faut construire/implmenter ces structures de donnes partir des possibilits de chaque langage.

Une implantation d’une structure de donne repose sur :

  • le choix d’une reprsentation mmoire,

  • le choix d’une implmentation algorithmique des oprations de la structure de donne.

  • deux lments fondamentaux prendre en compte : espace mmoire - temps excution/accs.

Dans les environnements de dveloppement, on trouve des bibliothques (packages java) qui offrent diverses implmentations des structures de donnes.

Par exemple, en java on trouve au moins deux implmentations de la structure de donne liste itrative :

  • ArrayList<E> : liste implante sous forme de tableau,

  • LinkedList<E> : liste implante sous forme de structure doublement chane.

On trouvera aussi :

  • Structure de donne pile :

    • Stack<E> : pile implante sous forme de tableau,

    • et d’autres …

  • Structure de donne file :

    • ArrayDeque<E> : file implante sous forme de tableau.

    • et d’autres …

  • Structure de donne ensemble (collection sans doublons)

    • HashSet<E> : ensemble implant sous forme de paires clef/valeur,

    • TreeSet<E> : ensemble implant sous forme d’arbre balanc.

  • Structure de donne map (association clef/valeur)

    • HashMap<K,V> : map implante sous forme d’une table de hachage,

    • TreeMap<K,V> : map implante sous forme d’arbre balanc.

3. Principe d’un "tableau dynamique"

Fonctionnement des tableaux en java :

  • Ils sont crs au run-time.

  • Une fois crs : la taille ne peut pas tre modifie.

  • Une variable tableau contient une rfrence du tableau (pas le tableau lui-mme).

4. Classe dvelopper : Pile de String

La pile :

  • Structure de donne de type LIFO (Last In, First Out).

  • Dernier lment "entr" dans la pile sera le premier "sorti".

On souhaite dfinir une classe Pile de String dont l’implmentation est un tableau "dynamique" de la taille exacte du nombre d'lments contenus dans la Pile. L’implantation propose ici consiste changer le tableau de stockage chaque modification de la pile.

Diagramme UML complet (gauche) et interface (droite) de la classe Pile Mthodes

Les mthodes :

* Pile() : Constructeur par dfaut qui construit une pile vide (sans lments).

* boolean estVide () : retourne true si la pile est vide, false sinon.

* String sommet () : donne accs au sommet de la pile, l'lment en "haut" de la pile (le dernier empil).

⇒ Prcondition : sommet() valide ssi pilevide() == false. Lve une exception PileException si non respect.

* void empiler (String pfElt) : permet d’ajouter un lment pfElt en "haut" de la pile.

* void depiler () : enlve l'lment en "haut" de la pile.

⇒ Prcondition : depiler() valide ssi pilevide() == false. Lve une exception PileException si non respect.

NB : noter que dans certains cas (livres, sites web, etc.), l’en-tte de depiler() pourrait tre String depiler (), signifiant que depiler() pourrait ET enlever le sommet de la pile ET renvoyer la valeur de ce dernier.
