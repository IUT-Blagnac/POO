2. Quelques explications

Pour les mthodes :

  1. Compte() : Constructeur non paramtr, appel aussi constructeur "par dfaut" : initialise un compte avec des dbits/crdits 0 euros et 0 oprations. Numro compte : "Pas de numro", propritaire : "Pas de propritaire".
  2. Compte(String pfNumCompte, String pfNomProp) : Constructeur paramtr : initialise un compte avec des (dbits/crdits) 0 euros et 0 oprations. Numro compte et propritaires sont passs en paramtre
  3. String getNumCompte() : permet de connatre le numro de compte.
  4. String getProprietaire() : permet de connatre (renvoyer) le nom du propritaire.
  5. double soldeCompte() : comme son nom l’indique : la somme restante sur le compte.
  6. void setProprietaire(String pfNomProp) : modifie le propritaire du compte.
  7. void retirer(double pfMontant) : permet de retirer un montant du compte. Elle lve une exception CompteException si montant < 0.
  8. void deposer(double pfMontant) : permet de dposer un montant sur le compte. Elle lve une exception CompteException si montant < 0.
  9. void afficher() : affiche l'cran this.toString() (redondante) Spcial pour les TPS uniquement
  10. String toString() : Permet d’obtenir la reprsentation String d’un compte. Construit une chaine contenant le numro de compte, le propritaire, le total des dbits, le total des crdits, le nombre d’oprations ralises.

    " Num. : aaa - Prop. : bbb - Debit xxx E / Credit yyy E / NbOps zzz"

3. Version 1

  1. crire la classe (le corps des mthodes) avec les attributs dfinis dans la page associe.
Solution Version 1
public class Compte {
        private String proprietaire;
        private String numCompte;
        private double debits, credits;
        private int nbDebits, nbCredits;

        public Compte() {
                this.numCompte = "Pas de numro";
                this.proprietaire = "Pas de propritaire";
                this.debits = 0;
                this.credits = 0;
                this.nbCredits = 0;
                this.nbDebits = 0;
        }

        public Compte(String pfNumCompte, String pfNomProp) {
                this.numCompte = pfNumCompte;
                this.proprietaire = pfNomProp;
                this.debits = 0;
                this.credits = 0;
                this.nbCredits = 0;
                this.nbDebits = 0;
        }

        public String getNumCompte() {
                return this.numCompte;
        }

        public String getProprietaire() {
                return this.proprietaire;
        }

        public double soldeCompte() {
                return this.credits - this.debits;
        }

        public void setProprietaire(String pfNomProp) {
                this.proprietaire = pfNomProp;
        }

        public void retirer(double pfMontant) throws CompteException {
                if (pfMontant < 0) {
                        throw new CompteException("Somme <0 sur retrait " + this.numCompte);
                } // Ou throw new CompteException();
                this.debits = this.debits + pfMontant;
                this.nbDebits++;
        }

        public void deposer(double pfMontant) throws CompteException {
                if (pfMontant < 0) {
                        throw new CompteException("Somme <0  sur dpot " + this.numCompte);
                } // Ou throw new CompteException();
                this.credits = this.credits + pfMontant;
                this.nbCredits++;
        }

        public void afficher() {
                System.out.println(this.toString());
        }

        public String toString() {
                return "  Num. : " + this.numCompte + " - Prop. : " + this.proprietaire
                                + " - Debit " + this.debits + " E / Credit " + this.credits
                                + " E / NbOps " + (this.nbCredits + this.nbDebits);
        }
}

7. Conclusion

C’est cela que l’on appelle l’encapsulation : Tant que l’interface de la classe est respecte par l’implmenteur et le client/utilisateur :

  • l’implmentation est compltement indpendante de l’utilisation,
  • l’utilisation est compltement indpendante de l’implmentation.

Le mcanisme utilis est :

  • les attributs sont TOUJOURS privs
  • l’interface (la liste des mthodes publiques) reste stable : identique en nombre de mthodes et signatures.
Exemples contradictoires

Si la classe tait dfinie sans attributs private :

package tps.banque;
import tps.banque.exception.CompteException;
public class Compte {
        public String proprietaire;
        public String numCompte;
        public double debit, credit;
        // Nb Oprations : compter depots et retraits
        public int nbDebits, nbCredits;
        // ...
}

Alors on peut crire le code suivant :

        Compte c ;
        c = new Compte("0101", "Jean"); // OK
        c.setProprietaire ("Marcel");
        c.nbDebits = c.nbDebits + 1; // On oublie d'enregistrer un dbit
        c.credit = c.credits - 100; // on crdite d'une valeur < 0
        c.afficher();

Cet exemple montre que :

  • Sans attributs private, on PEUT faire n’importe quoi ;-).
  • Le type Compte n’est pas "stable" ou "cohrent". Le programmeur peut rendre le compte "faux" : ce n’est plus la ralit. Tous tests possibles de la classe Compte n’y changeront rien …
  • Si on modifie la classe Compte comme nous l’avons fait en implmentation 2 … le code ne fonctionne plus (c.nbDebits = c.nbDebits + 1;).
