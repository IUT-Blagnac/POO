3. Test du polymorphisme

3.1. Préparation : ajout de méthodes

Questions :

  1. Dans la classe Compte

    1. Déclarer une nouvelle méthode :

      1. public void afficherAvecSolde ().

      2. Elle affiche le Compte (méthode this.afficher()) et ajoute le solde à l'écran (this.soldeCompte()).

    2. Déclarer une autre méthode :

      1. public void transferer (Compte c, double pSomme) throws CompteException.

      2. Elle transfère (virement) pSomme euros de l’objet courant (retrait, this.retirer(pSomme)) vers le compte c (dépôt, c.deposer(pSomme)).

      3. Elle lève une exception CompteException si le dépôt ou le retrait sur un des deux comptes ne peut pas avoir lieu normalement. En fait elle "laisse" passer les exceptions levées par retirer() ou deposer().

3.2. Premiers tests de polymorphisme

Questions :

  1. Ouvrir le fichier ClassEssai.java.

  2. Il contient la déclaration de trois variables de type Compte : cUn, cDeux, cTrois.

  3. Sans changer ces déclarations :

    1. A l’aide de cUn, créer un objet de type CompteBancaire (avec le constructeur paramétré).

    2. A l’aide de cDeux, créer un objet de type CompteEpargne (avec le constructeur paramétré).

    3. A l’aide de cTrois, créer un objet de type CompteMoneo (avec le constructeur paramétré).

  4. Déposer 2000 € sur chacun des trois comptes.

  5. Appliquer la méthode afficherAvecSolde() à chacun de ces trois comptes. Quelle est la méthode afficher() appelée pour chacun d’eux ?

  6. Transférer 500 € de cUn vers cDeux.

  7. Transférer 500 € de cUn vers cTrois.

  8. Appliquer la méthode afficherAvecSolde() à chacun de ces trois comptes. Quelle est la méthode afficher() appelée pour chacun d’eux ?

  9. Remarquez que, depuis le début de cette partie :

    1. Vous manipulez des variables déclarées de type Compte (type statique).

    2. Qui au "runtime" référencent un CompteBancaire, un CompteEpargne, un CompteMoneo.

    3. Et réagissent en fonction de leur type dynamique.

  10. Déclarer une variable tabC de type tableau de Compte.

  11. Créer avec tabC un tableau de 3 Compte (références vers des objets Compte).

  12. Stocker :

    1. cUn dans l'élément 0 de tabC.

    2. cDeux dans l'élément 1 de tabC.

    3. cTrois dans l'élément 2 de tabC.

  13. Par une boucle for, appliquer la méthode afficherAvecSolde() à chacun des éléments du tableau. Quelle est la méthode afficher() appelée pour chacun d’eux ?

  14. Par une boucle for, déposer 3000 € sur chacun des comptes de tabC.

  15. Transférer 1000 € de tabc[0] vers tabc[1].

  16. Transférer 1000 € de tabc[0] vers tabc[2].

  17. Par une boucle for, appliquer la méthode afficherAvecSolde() à chacun des éléments du tableau. Quelle est la méthode afficher() appelée pour chacun d’eux ?

4. Le polymorphisme en vraie grandeur

L’objectif maintenant est d’utiliser plusieurs types de comptes définis dans la classe AgenceBancaire. Pour cela nous allons modifier la classe déjà développée ClasseApplicationAgenceBancaire.java.

Questions :

  1. Préparer l’activité :

    1. Dans le même répertoire que précédemment, récupérer les classes du serveur tp11-partie2. Ce fichier contient :

      1. ClasseApplicationAgenceBancaire.java : classe à modifier.

      2. LanceurApplication.java : programme contenant le main() à lancer.

      3. AgenceBancaire.java ainsi que les différentes classes d’exceptions nécessaires ABCompteDejaExistantException, ABCompteInexistantException, ABCompteNullException.

    2. Ouvrir le fichier ClasseApplicationAgenceBancaire.java.

  2. Dans ce TP, on considérera que :

    1. AgenceBancaire est la classe Modèle

    2. ClasseApplicationAgenceBancaire est la classe regroupant à la fois Vue et Contrôleur.

    3. LanceurApplication est la classe permettant de lancer l’application sur une agence particulière.

  3. Étudier le code de la classe ClasseApplicationAgenceBancaire.

    1. Par rapport à une version antérieure de TP, la classe a évolué et elle définit maintenant des objets mettant en oeuvre l’application de gestion des comptes sur un objet AgenceBancaire passé en paramètre du constructeur.

    2. Etudier le constructeur

    3. Etudier la méthode runApplication().

    4. Etudier les méthodes voirUnCompte (), supprimerUnCompte(), deposerSurUnCompte(), retirerSurUnCompte().

  4. Lancer l’application, elle doit fonctionner mais les fonctions "Créer un compte" (monéo, bancaire, épargne, jeune) et "Transférer" ne sont pas implémentées.

  5. Écrire les méthodes :

    1. private void creerCompteBancaire () pour i) saisir les paramètres nécessaires ii) créer un compte bancaire et iii) l’ajouter à l’agence bancaire (cf. attribut monAg). Utiliser le constructeur paramétré le plus complet.

    2. private void creerCompteEpargne () pour i) saisir les paramètres nécessaires ii) créer un compte épargne et iii) l’ajouter à l’agence bancaire (cf. attribut monAg). Utiliser le constructeur paramétré le plus complet.

    3. private void creerCompteMoneo () pour i) saisir les paramètres nécessaires ii) créer un compte monéo et iii) l’ajouter à l’agence bancaire (cf. attribut monAg). Utiliser le constructeur paramétré le plus complet.

    4. private void creerCompteJeune () pour i) saisir les paramètres nécessaires ii) créer un compte jeune et iii) l’ajouter à l’agence bancaire (cf. attribut monAg). Utiliser le constructeur paramétré le plus complet.

  6. Écrire la méthode :

    1. private void transfertEntreDeuxComptes () permettant de réaliser un transfert : i) saisir le numéro de deux comptes, ii) saisir un montant, iii) transférer le montant saisi du premier compte vers le second.

  7. Tester la création de compte, le dépôt de sommes sur les comptes, le retrait, le transfert entre comptes, l’affichage de l’AgenceBancaire, l’affichage individuel des comptes. Tout doit fonctionner …

Questions - Analyse du code écrit :

  2. Analyser le code de ClasseApplicationAgenceBancaire. A quels "moments" précis se pose-t-on la question du type de compte réellement manipulé ?

  3. Analyser le code de ClasseApplicationAgenceBancaire. Quel type manipulent les méthodes voirUnCompte(), supprimerUnCompte(), deposerSurUnCompte(), retirerSurUnCompte ?

  4. Analyser le code de ClasseApplicationAgenceBancaire. Quel type manipule la méthode transfertEntreDeuxComptes() ?

5. Pousser le polymorphisme plus loin

Question - Classes abstraites :

  1. On peut constater que les classes Compte, CompteDepot, et CompteSecurise ne représentent pas des comptes réels que les banquiers proposent à leurs clients.

  2. Déclarer abstract les classes Compte, CompteDepot, et CompteSecurise. Le programme continue-t-il à fonctionner ?

Question - Méthodes spécifiques :

  1. Pour les classes héritant de CompteSecurise (CompteBancaire, CompteJeune), on ne peut pas utiliser dans ClasseApplicationAgenceBancaire la méthode isRetraitPossible(). Essayer d’appeler isRetraitPossible() dans dans la méthode de retrait sur un Compte retirerSurUnCompte() pour vérifier. Expliquez pourquoi.

  2. Déclarer la méthode isRetraitPossible() comme abstract dans Compte.

  3. On souhaite que la méthode isRetraitPossible() soit abstract dans CompteDepot. Faut-il écrire quelque chose ? Quoi ? Rien ?

  4. La redéfinir si besoin et correctement dans les classes CompteMoneo, CompteEpargne, CompteBancaire, CompteCompteJeune. A noter : jusqu'à maintenant, tout retrait est possible quelque soit son montant (>=0) sur les comptes monéo et épargne.

  5. Utiliser la méthode isRetraitPossible() pour tester un retrait avant de l’effectuer dans la classe ClasseApplicationAgenceBancaire dans les méthodes : retirerSurUnCompte() et transfertEntreDeuxComptes().

  6. Tester l’application et essayer des dépôts sur les 4 types de comptes et des transferts entre les 4 types de comptes.

Conclusion :

  • Pour faciliter le polymorphisme : les méthodes sont à déclarer "le plus haut possible" dans une hiérarchie d’héritage.

  • Ce point est un problème de conception des classe.
